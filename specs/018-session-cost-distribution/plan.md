# Implementation Plan: Session Cost Calculation & Part Cost Distribution

**Branch**: `018-session-cost-distribution` | **Date**: 2026-04-01 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/018-session-cost-distribution/spec.md`

## Summary

Extend the existing two-step cutting session creation modal to track cost at batch-level granularity. Step 1 replaces the simple meters input with a per-batch consumption table (one row per inbound purchase) and adds live employee cost and session cost preview displays. Step 2 adds a per-batch pattern for consumed materials and a cost distribution table that lets the user assign unit costs to produced part rows, with auto-proportional adjustment and a locked/auto state per row. On submission, all batch consumptions are persisted individually, and cost figures are stored on the session record and propagated to each produced piece.

```sql
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  stock_transaction_id TEXT NOT NULL,

```

**Backward compatibility**: All new columns are nullable. Existing sessions display NULL cost columns as 0.00 in the UI. No data loss or integrity issues.

### Phase B: Backend — New IPC Handlers

**File**: `electron/main.js`

#### B1 — `cutting:getFabricBatches`

```
Input:  { stockItemId: string, color: string }
Output: FabricBatch[]

Query:
  SELECT
    st.id AS transactionId,
    st.transaction_date AS transactionDate,
    st.price_per_unit AS pricePerMeter,
    st.quantity AS originalQuantity,
    st.supplier_name AS supplierName,
    (st.quantity - COALESCE(SUM(cbc.quantity), 0)) AS availableQuantity
  FROM stock_transactions st
  LEFT JOIN cutting_batch_consumptions cbc ON cbc.stock_transaction_id = st.id
  WHERE st.stock_item_id = ? AND st.color = ? AND st.type = 'inbound'
  GROUP BY st.id
  ORDER BY st.transaction_date DESC
```

#### B2 — `cutting:getMaterialBatches`

```
Input:  { stockItemId: string, color?: string | null }
Output: MaterialBatch[]

Same query pattern as B1, with color filter applied only when color is non-null.
Join stock_items to get unit field.
```

#### B3 — Extend `cutting:create`

Add to the existing transaction block:

1. **Pre-insert validation** (server-side, re-computes available quantities):
   - For each `fabricBatchConsumption`: re-run the available-quantity query; reject if `quantity > availableQuantity`
   - For each `materialBatchConsumption.batches`: same check

2. **Derive `metersUsed`**: `SUM(fabricBatchConsumptions[].quantity)` → store in `cutting_sessions.meters_used` (backward compat)

3. **Insert `cutting_batch_consumptions` rows**:
   - One per fabric batch (is_fabric=1)
   - One per material batch entry (is_fabric=0)

4. **Store cost fields on cutting_sessions**:
   - `fabric_cost`, `employee_cost`, `consumed_materials_cost`, `total_session_cost`

5. **Store `unit_cost` on `cutting_session_parts`**:
   - Match each `partCosts[]` entry to the inserted `cutting_session_parts` row by `(partName, sizeLabel)`

6. **Store `unit_cost` on `cutting_pieces`**:
   - Each inserted piece inherits `unit_cost` from its `cutting_session_parts` match

7. **Keep existing consumed logic**:
   - Still insert into `cutting_consumption_entries` (total per item+color)
   - Still insert into `stock_transactions` (type='consumed', total per item+color) for stock tracking

#### F4 — `CostDistributionTable.tsx`

**File**: `frontend/components/cutting/CostDistributionTable.tsx`

**Props**:
```typescript
interface Props {
  parts: PartRow[];                       // from the parts editor above
  totalSessionCost: number;
  rows: CostDistributionRow[];            // controlled
  onChange: (rows: CostDistributionRow[]) => void;
  error: string | null;                   // all-locked mismatch error
}
```

**Renders**:
- Header: "توزيع التكلفة على الأجزاء — التكلفة الإجمالية: {totalSessionCost} دج"
- Table columns: الجزء | المقاس | العدد | التكلفة/قطعة | الإجمالي
- Each row: part name, size (read-only) | count (read-only) | unit cost input with "auto" / "محدد" badge | row total (read-only)
- Grand total footer: "المجموع: {grandTotal} دج / المطلوب: {totalSessionCost} دج" — green if match, red if mismatch
- Error banner (below table, red) when all rows locked and mismatch

**Internal logic**:
- On mount / when `parts` or `totalSessionCost` changes: recalculate default unit costs (equal distribution) for all 'auto' rows
- On unit cost edit: mark row as 'locked'; redistribute remaining session cost proportionally among 'auto' rows
- Redistribute algorithm:
  1. Compute `lockedTotal = SUM(locked rows: unitCost × count)`
  2. Remaining = `totalSessionCost − lockedTotal`
  3. For each auto row: `unitCost = Remaining / totalAutoCount` (equal, not proportional by count)
  4. Last auto row gets `Remaining − SUM(other auto rows: unitCost × count)` to absorb rounding

*(Note: "proportionally" in the spec means proportional to count — equal per-piece distribution among auto rows)*

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Inbound batches may have `price_per_unit = NULL` if imported from older data | Display as 0.00; exclude from cost computation; show a warning badge on that batch row |
| User selects a batch and another session is submitted simultaneously (race condition) | Backend re-validates availability inside the SQLite transaction; returns error if overdraw detected |
| `CostDistributionTable` receives parts with duplicate (partName, sizeLabel) combos | `partCosts` matching uses first-match; spec assumes uniqueness from the parts editor above |
| Rounding accumulation over many auto rows | Last-row absorption strategy guarantees grand total = session cost exactly, regardless of row count |
