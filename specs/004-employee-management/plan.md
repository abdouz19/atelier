## Summary

Add an Employee Management module to the Atelier desktop app. The feature covers three user stories: (1) an employee directory with add, edit, and status toggle (deactivate/re-activate); (2) a per-employee detail view showing the full operations history grouped by type with financial aggregates (total earned, total paid, balance due); and (3) payment logging with full mutability (add, edit, delete). Employees also support manual operation record entry from the detail view. The implementation follows the established Electron + Next.js IPC architecture: Drizzle schemas → SQLite migrations → queries → service → IPC handler → preload → ipc-client → hooks → components → page.

## Technical Context

**Language/Version**: TypeScript 5 (strict mode) — Electron main process + Next.js 14 App Router renderer
**Primary Dependencies**: Electron, Next.js 14 (static export), Drizzle ORM (better-sqlite3), react-hook-form + Zod, shadcn/ui + Tailwind CSS, Zustand
**Storage**: SQLite via Drizzle ORM — three new tables: `employees`, `employee_operations`, `employee_payments`
**Testing**: Not requested for this feature
**Target Platform**: macOS/Windows desktop (Electron)
**Project Type**: Desktop app (Electron + Next.js static export)
**Performance Goals**: Detail view with 1,000 operation records loads in ≤ 2 s (SC-002); balance recalculates within 1 s of payment submission (SC-003)
**Constraints**: Static export (`output: 'export'`) — no server-side routes; detail views use `?id=` query-param pattern; 150-line component limit (constitution)
**Scale/Scope**: Single-user desktop app; expected O(100s) of employees, O(1,000s) of operations per employee

## Key Implementation Details

### Photo Handling
Reuses the exact pattern from `electron/features/stock/stock.service.ts`:
- `getPhotosDir()` → `app.getPath('userData')/employee-photos/`
- `savePhoto(base64, mimeType)` → writes file, returns filename
- `deletePhoto(filename)` → removes file from disk
- `resolvePhotoPath(filename | null)` → returns `file:///...` absolute path or null

### Balance Due Computation
Computed with SQL subqueries embedded in Drizzle selects (never stored). This ensures FR-019 (always current) is satisfied without manual refresh:
```sql
-- embedded in getAll:
(SELECT COALESCE(SUM(total_amount), 0) FROM employee_operations WHERE employee_id = employees.id)
- (SELECT COALESCE(SUM(amount), 0) FROM employee_payments WHERE employee_id = employees.id)
AS balance_due
```

### Operations Grouping
`getById` query returns flat `OperationRecord[]` ordered by `operation_date DESC`. The service `getById` function groups rows by `operation_type` and computes per-group `subtotal` and `count` before returning the `EmployeeDetail` DTO.

### Mutation Channels Return Full Detail
`addOperation`, `addPayment`, `updatePayment`, and `deletePayment` all return the updated `EmployeeDetail` directly. The `useEmployeeDetail` hook replaces its state with the returned detail — no second IPC call needed (satisfies SC-003).

### Employee Selectors (Cross-App)
When future production modules (cutting, distribution, QC, finition) need an employee selector, they must:
1. Call `employees:getAll` and filter client-side to `status === 'active'`, OR
2. Add a dedicated `employees:getActive` channel.
The plan for this feature is approach 1 (filter in component) since no production modules are being built yet.

### Component Sizing
The detail page is the most complex view. It is split into:
- `EmployeeDetailPanel` (orchestrates layout — ≤ 150 lines)
- `FinancialSummaryCard` (financial totals — standalone, ≤ 80 lines)
- `OperationsSummaryCard` (per-type summary cards — ≤ 80 lines)
- `OperationsHistory` (operation groups + add button — ≤ 150 lines)
- `PaymentsHistory` (payment rows + edit/delete/add — ≤ 150 lines)
- `EmployeeDetailView` (loading/error shell — ≤ 40 lines)

### Status Toggle UX
- Deactivate: action button + `ConfirmDialog` (required by constitution + spec)
- Re-activate: action button, no confirmation (FR-004 from clarification Q2 answer A)
- The action button label changes based on current status: "تعطيل" when active, "تفعيل" when inactive
